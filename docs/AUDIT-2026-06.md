# Fittin.be — Code & Site Audit (juni 2026)

Volledige read-only audit door 13 parallelle reviewers (bevindingen adversarieel geverifieerd tegen de echte code).
**Totaal: 61 bevestigde bevindingen** — 3 kritiek · 18 hoog · 18 middel · 22 laag.

Elke bevinding: bestand:regel · impact · voorgestelde fix.


## 🔴 KRITIEK (3)

### 1. Stripe webhook marks event processed before handling and swallows handler errors → permanent loss of credits/booking confirmation
- **Categorie:** data-integrity · **Gebied:** booking-payments
- **Locatie:** `app/api/stripe/webhook/route.js:31-39`
- **Impact:** If handleEvent throws transiently (DB blip, RPC error, email/import failure surfacing as an exception, etc.) the side effect — grantCredits for a punch card, markBookingPaid, membership upsert, coach-credit top-up — never completes. Because event.id is already in stripe_events, Stripe's automatic retry is deduped at line 32 and discarded. Result: the customer paid in Stripe but received no credits / their booking stays unpaid / membership not activated, with no recovery path.
- **Fix:** Only mark the event processed AFTER the handler succeeds: run handleEvent first, then insert into stripe_events (or wrap insert+handler in a single transaction and roll back on failure). On handler error, return a non-2xx status so Stripe retries. Use the pre-insert pattern purely as a best-effort lock you DELETE on failure, or move dedup to an upsert that only sticks once the work commits.

### 2. bookings_update RLS lets a member mark their own booking paid=true (Stripe payment bypass)
- **Categorie:** security · **Gebied:** rls-schema
- **Locatie:** `supabase/migrations/0001_init.sql:352-353`
- **Impact:** The policy authorises a member to UPDATE ANY column of their own booking rows. Unlike profiles (locked in 0015 via `revoke update` + column GRANT + WITH CHECK), bookings has no column-level GRANT restriction and no BEFORE UPDATE guard trigger (the only bookings triggers are reject_blocked/free_expired on INSERT, refund_coach_credit and post_on_booking). The legitimate writer of `paid=true` is the Stripe webhook (app/api/stripe/webhook/route.js:80, service role). A member holding the public anon key can call `supabase.from('bookings').update({ paid: true, price_cents: 0, payment_source: 'credit' }).eq('id', myUnpaidBooking)` directly, confirming a real €11/€15 reservation (and door access via open_door()) for free. expire_unpaid_bookings only cancels rows with paid=false, so flipping paid=true also makes the slot permanent. The 0052 trigger even posts an 'activity' to the feed on paid=true. This is a direct payment/revenue-loss hole.
- **Fix:** Lock the columns members may change, mirroring 0015. Add `revoke update on bookings from authenticated, anon; grant update (status, cancelled_at, notes) on bookings to authenticated;` (members only cancel / annotate), OR replace the member self-update path with a SECURITY DEFINER cancel_booking() RPC and remove the member branch of bookings_update. Keep the staff branch. Ensure paid/price_cents/payment_source/starts_at can only be written by SECURITY DEFINER RPCs and the service role.

### 3. Overlapping (multi-hour) bookings are not prevented at the DB level — only a TOCTOU EXISTS check
- **Categorie:** data-integrity · **Gebied:** rls-schema
- **Locatie:** `supabase/migrations/0043_multi_hour.sql:54-58`
- **Impact:** Since 0043 introduced 1–4h bookings, the partial unique index from 0001 (keyed on starts_at only) no longer enforces exclusivity for ranges that overlap but start at different hours. Overlap is enforced only by the in-function EXISTS check, which is non-atomic under READ COMMITTED: two concurrent create_booking calls (e.g. 14:00–16:00 and 15:00–17:00) each see no conflict, insert with different starts_at (unique index does not fire), and both commit — double-booking the 'exclusive private gym'. Worse, admin_create_booking (0025) and coach_book_session (0049) perform NO overlap check at all and rely solely on the same-start unique index, so a coach/admin 60-min booking that lands inside an existing member's multi-hour booking (different start hour) is silently accepted. Breaks the core privégym invariant.
- **Fix:** Add a true range-exclusion constraint so the DB rejects any overlap atomically: `create extension if not exists btree_gist;` then `alter table bookings add constraint bookings_no_overlap exclude using gist (gym_id with =, tstzrange(starts_at, ends_at) with &&) where (status = 'bevestigd');` Then catch exclusion_violation in all booking RPCs (create_booking, admin_create_booking, coach_book_session). This makes double-booking impossible regardless of start hour and removes the race; the per-start unique index can stay as a cheap fast-path or be dropped.

## 🟠 HOOG (18)

### 4. Cancelling a credit-paid booking never returns the session credit (member or admin cancel)
- **Categorie:** data-integrity · **Gebied:** admin-domain
- **Locatie:** `app/beheer/actions.js:96-106`
- **Impact:** When a booking paid with a session credit is cancelled — even by the member within the free 24h window, or by an admin comping the cancel — the deducted credit is gone forever. This directly contradicts the business rule 'Cancel free up to 24h before' and quietly steals paid-for sessions from members. Affects every credit-based booking cancellation.
- **Fix:** Add an AFTER UPDATE trigger on bookings mirroring refund_coach_credit: when new.status='geannuleerd' and old.status<>'geannuleerd' and payment_source='credit' (and, for the strict rule, the cancel happened >= gym.cancel_hours before starts_at), insert credits_ledger (gym_id, user_id, +N, 'refund', booking_id) once (guard via NOT EXISTS reason='refund' ref_id=id). N should match the hours/credits originally deducted (see 0043/0050 multi-hour deduction).

### 5. A coach can approve/delete events via approveEvent (requireStaff, not beheerderOnly) — bypasses the admin approval gate
- **Categorie:** role-boundary · **Gebied:** admin-domain
- **Locatie:** `app/beheer/community-actions.js:40-93`
- **Impact:** The entire 0031 event-approval workflow is bypassable by the very coaches it is meant to gate: a coach can publish their own events (any price/capacity, blocking gym slots for paying sessions) without admin sign-off, and can delete/alter community events. Defeats the documented approval control.
- **Fix:** Change approveEvent (and arguably createEvent/deleteEvent/createChallenge/deleteChallenge, which are admin pages) to `await requireStaff(true)` so only beheerders can approve or manage community content. Keep coach event creation in app/coach which already forces status:'pending'.

### 6. Activation campaigns re-grant reward_credits to the same member every cooldown period (unbounded free sessions)
- **Categorie:** data-integrity · **Gebied:** admin-domain
- **Locatie:** `lib/activation.js:82-135`
- **Impact:** Free session credits compound for members who never actually book — e.g. a 'never booked, here are 2 free sessions' campaign keeps stacking 2 credits/month onto inactive accounts, inflating liabilities and letting someone bank many free sessions by simply never matching out of the segment. Real monetary leak.
- **Fix:** Before crediting, exclude members who already have a credits_ledger row for this campaign (add ref_id=campaign_id and check NOT EXISTS), or cap rewards to once per (campaign,user). At minimum gate the reward grant on first send rather than every send, separate from the email cooldown.

### 7. Cron and queue/process endpoints are world-callable when CRON_SECRET is unset (fail-open auth)
- **Categorie:** security · **Gebied:** api-security
- **Locatie:** `app/api/queue/process/route.js:17-21`
- **Impact:** Both /api/queue/process and /api/cron/activation gate auth behind `if (secret)`. If CRON_SECRET is not configured in the Vercel project (it is NOT a required/validated env var anywhere — see lib/supabase/config.js style validation only covers Supabase), the guard is skipped entirely and ANY anonymous visitor can hit these GET endpoints. /api/queue/process drains the newsletter send-queue via the service-role client (processSendQueue uses createAdminClient and resend.batch.send), so an attacker can force-send queued campaigns / burn the Resend quota. /api/cron/activation runs expire_unpaid_bookings, sends reminder + activation emails, and re-triggers the queue. Same fail-open pattern at app/api/cron/activation/route.js:16-22.
- **Fix:** Make the secret mandatory: if (!secret) return 503/500 'cron not configured', and always require the header to match. Prefer Vercel's built-in cron auth (verify the `x-vercel-cron` / `Authorization: Bearer ${CRON_SECRET}` Vercel injects) and reject when the env var is missing rather than silently allowing access.

### 8. FittinWelcome free session not gated on welcome_status server-side — card-fingerprint anti-farm gate is bypassable
- **Categorie:** role-boundary · **Gebied:** booking-payments
- **Locatie:** `supabase/migrations/0050_pt_coach_price.sql:23, 45`
- **Impact:** A user who never activated their card (welcome_status still 'unclaimed', or 'blocked' because their card already farmed a free session on another account) can call the server action / RPC with useWelcome=true and still get a free Fit60 session. The fingerprint dedupe in handleWelcomeSetup is defeated because the booking path doesn't require the eligible state. Free sessions can be farmed across throwaway accounts, exactly what 0009 was built to prevent.
- **Fix:** Enforce eligibility in the security-definer RPC: read welcome_status alongside welcome_code_used and compute `v_free := p_use_welcome and not coalesce(v_used,false) and welcome_status = 'eligible' and v_srv.type='fit60' and v_hours=1`. (Belt-and-suspenders: also re-check welcome_status in createBookingAction before calling the RPC.)

### 9. 6-month package credit expiry is advertised but never enforced — credits never expire
- **Categorie:** data-integrity · **Gebied:** booking-payments
- **Locatie:** `app/api/stripe/webhook/route.js:62-75`
- **Impact:** The 10-session beurtenkaart is sold as '6 maanden geldig' (lidmaatschap/page.jsx:106 and the footer line 132 'Sessies worden direct bijgeschreven na betaling (6 maanden geldig)'), but a member can spend those credits forever. Revenue model / accounting mismatch and a customer-facing promise the system silently breaks (in the customer's favour, but the books and any audit will be wrong).
- **Fix:** Either set credits_ledger.expires_at on the grant and make balance = sum(delta) WHERE expires_at IS NULL OR expires_at > now() everywhere (RPC + UI), or run a scheduled job that, when a punch_card expires, inserts a negative 'verval' adjustment row for the unused remainder. Pick one source of truth and apply it in create_booking, admin_create_booking, and all UI balance reads.

### 10. Paid events can be oversold: capacity checked only against paid signups, webhook confirms without re-checking
- **Categorie:** data-integrity · **Gebied:** booking-payments
- **Locatie:** `app/(site)/community/actions.js:33-34`
- **Impact:** For the last seat(s), N users can each open checkout simultaneously (all see taken < capacity because none are paid yet), all pay, and the webhook marks all paid=true. The event is oversold and every overbooked customer was charged for a spot that doesn't exist. Refund/ops headache and broken event flow.
- **Fix:** Re-check and reserve capacity atomically at confirmation time. Move the seat allocation into a security-definer RPC / DB constraint: e.g. a partial count guard inside a transaction in the webhook before flipping paid=true, refunding via Stripe if the event is now full; or a DB trigger that rejects the (paid=true) update when paid count would exceed capacity, and surface that to refund logic.

### 11. A coach can book sessions for (and spend credits of) members who are not their clients
- **Categorie:** role-boundary · **Gebied:** coach-domain
- **Locatie:** `supabase/migrations/0049_coach_credits.sql:49-50`
- **Impact:** Any coach can create real room bookings for arbitrary gym members (occupying the slot, sending them 'your coach booked a session' emails/notifications), bill the gym for room time against their own coach_ledger, and — with the 'Gebruik sessietegoed van client' checkbox (page.jsx:267) — decrement the prepaid coach_credit_ledger balance of a member who is not their client. Contrast sendCoachPaymentRequest (actions.js:241-242) which DOES verify the link; booking does not.
- **Fix:** In coach_book_session add: `if not exists (select 1 from coach_clients where coach_id = v_coach and client_id = p_client and gym_id = v_gym) then raise exception 'Dit is niet jouw client.'; end if;` (skip for role 'beheerder' if admins should book anyone). Also restrict the dashboard/scheduler member list to coach_clients-linked clients for coaches, mirroring app/coach/clienten/page.jsx:17-21.

### 12. Referred friend receives 2 free-session credits instead of the 1 promised
- **Categorie:** data-integrity · **Gebied:** community-notify
- **Locatie:** `supabase/migrations/0051_referral_rewards.sql:12-13`
- **Impact:** Every referred friend ends up with 2 free sessions (€22 of value) while the UI on /community promises exactly one ('cadeauen we hen een gratis sessie'). Each successful referral leaks an extra free session; at scale this is a recurring revenue loss and the leaderboard/credits balance no longer matches the stated rules.
- **Fix:** Pick a single moment to grant the friend's credit. Either remove the immediate `reason='referral'` credit in 0021's redeem_referral (grant only at activation via 'referral_welcome'), or remove the 'referral_welcome' insert in 0051 and keep the redeem-time credit. Given the spec says reward 'after the friend actually pays', prefer crediting only in reward_pending_referral and drop the redeem-time credit_ledger insert.

### 13. Queued newsletter still sends to subscribers who unsubscribe before the queue drains
- **Categorie:** data-integrity · **Gebied:** community-notify
- **Locatie:** `lib/newsletter.js:125-152`
- **Impact:** A large blast is queued instantly, then drained over many minutes. Anyone who clicks Uitschrijven during that window is still emailed the campaign, directly contradicting the unsubscribe promise on /uitschrijven and the List-Unsubscribe header. This is a compliance/trust problem, not just UX.
- **Fix:** In processSendQueue, join subscriber status and skip/mark-failed any row whose subscriber is no longer 'active' before building the Resend payload (e.g. select 'id, unsub_token, status' and filter `s.status === 'active'`; for the rest, set campaign_sends.status='failed' or 'unsubscribed'). Alternatively, have unsubscribeByToken also flip that subscriber's still-'queued' campaign_sends to 'failed'.

### 14. Resend bounce/complaint events never mark the subscriber bounced/unsubscribed
- **Categorie:** data-integrity · **Gebied:** community-notify
- **Locatie:** `lib/newsletter.js:244-275`
- **Impact:** Hard-bounced and complaining recipients keep being targeted by every future campaign and drip. Repeatedly mailing complainers and dead addresses tanks the sender reputation of news.fittin.be and can get the domain blocked — a real deliverability/legal (complaint-handling) risk, not cosmetic.
- **Fix:** On 'email.bounced'/'email.complained', look up the subscriber via the matching campaign_sends.subscriber_id (or email+gym_id) and set subscribers.status = 'bounced' (or 'unsubscribed' for complaints). This permanently suppresses them from future active-subscriber queries.

### 15. profiles_public_coaches exposes full coach row (email, phone, stripe_customer_id) to anon
- **Categorie:** security · **Gebied:** rls-schema
- **Locatie:** `supabase/migrations/0036_coach_profiles.sql:9-11`
- **Impact:** RLS grants row-level, not column-level, access. This SELECT policy is OR'd with profiles_select and matches for ANY caller including anon, so anyone with the public anon key (shipped in the browser bundle) can run `supabase.from('profiles').select('email, phone, stripe_customer_id, pending_referral').eq('role','coach').eq('coach_public', true)` and read every public coach's private contact data and Stripe customer id. coaches are auto-set coach_public=true on promotion (app/beheer/actions.js:346), so all three coaches (Yoshe, Jelle, Billy) are exposed. The app's own /coaches pages avoid this by querying via the service-role admin client with explicit public columns, but the policy itself is the security boundary and it over-shares.
- **Fix:** Do not expose the base profiles table publicly. Either (a) move the public-facing coach fields (full_name, coach_bio, coach_specialty, coach_photo_url, coach_pricelist, coach_pt_price_cents) into a dedicated coach_public_profiles table / view with a SELECT-true policy and drop profiles_public_coaches, or (b) keep serving /coaches exclusively through the service-role client (as the code already does) and drop the anon-readable policy entirely. A column-restricted RLS policy is not possible in Postgres, so a view/separate table is the correct fix.

## 🟡 MIDDEL (18)

### 22. Challenge reward credits are never actually awarded to members
- **Categorie:** bug · **Gebied:** admin-domain
- **Locatie:** `app/beheer/community-actions.js:15-30`
- **Impact:** Every challenge created in /beheer/challenges promises members '+N gratis sessies' (shown on the public community page and admin UI) but completing the challenge never adds a single credit. Members who hit the goal are silently never rewarded — a broken core community feature and a reputational/trust problem since the gym publicly advertises the reward.
- **Fix:** Add the missing crediting path: either a DB function run from the activation cron (app/api/cron/activation/route.js) that, for each active challenge, finds members whose confirmed-booking count in [starts_on, ends_on] >= goal_count and who don't yet have a credits_ledger row with reason='challenge' + ref_id=challenge_id, inserting (gym_id, user_id, +reward_credits, 'challenge', challenge_id); or award on booking-confirm. Use the challenge_progress table (already in schema) with reached_at to dedupe so each member is rewarded once per challenge.

### 23. Generated invoices use a truncated UUID/ID slice, not a unique consecutive number (gyms.invoice_seq unused)
- **Categorie:** content · **Gebied:** admin-domain
- **Locatie:** `app/beheer/factuur/page.jsx:42`
- **Impact:** Belgian VAT law requires invoices to carry a unique, consecutive sequence number. A first-8-hex-chars slice is neither sequential nor guaranteed unique (8 hex chars can collide), and the coach invoice number is identical for every coach+month regardless of content. These invoices are not legally compliant and could be rejected by accounting/FOD Financiën.
- **Fix:** Use the gyms.invoice_seq counter: atomically increment it (e.g. an RPC `update gyms set invoice_seq=invoice_seq+1 returning invoice_seq`) and format like 2026-0001 when an invoice is finalized, persisting the assigned number on the payment/booking so re-renders are stable. Don't derive the number from a UUID slice.

### 24. Resend stats and inbound-email webhooks accept unsigned requests when their secret is unset
- **Categorie:** security · **Gebied:** api-security
- **Locatie:** `app/api/resend/webhook/route.js:14-16`
- **Impact:** verifySignature returns true (accept) when RESEND_WEBHOOK_SECRET is unset; the identical pattern is in app/api/inbound/email/route.js:16-18 (`const secret = process.env.RESEND_INBOUND_SECRET; if (!secret) return true;`). If either secret is not set in production, anyone can POST forged events. For inbound/email a forged {type:'email.received', data:{email_id}} causes importReceived() to run with the service-role client and fetch an arbitrary Resend message id into inbound_emails; for resend/webhook a forged event lets an attacker manipulate campaign delivered/opened/clicked/bounced counters (recordResendEvent writes via service-role). Defaulting verification to 'open' is the wrong default for an unauthenticated, service-role-backed endpoint.
- **Fix:** Fail closed: when the corresponding secret is unset, return 401/503 instead of `return true`. Only skip verification behind an explicit, separate dev-only flag — never as the default for a missing secret.

### 25. Discount code redemption is recorded and used_count bumped before payment — abandoned checkouts burn a per-user-once code
- **Categorie:** bug · **Gebied:** booking-payments
- **Locatie:** `app/(site)/boeken/actions.js:123-126`
- **Impact:** A member who applies a per_user_once / max_uses code, gets redirected to Stripe, then cancels or abandons checkout has permanently consumed the code (validateDiscount line 14-16 will then reject it: 'Je hebt deze code al gebruikt.'). For activation/win-back campaigns this silently kills the discount the member never actually used, and prematurely exhausts max_uses pools.
- **Fix:** Record the redemption in the webhook on successful payment (the discount_code_id is already passed in the checkout metadata at boeken/actions.js:42), not at session-creation time. Make recordRedemption idempotent on the Stripe session/booking id so replays don't double-count.

### 26. resumeCheckoutAction re-charges full price, silently dropping any discount the original booking had
- **Categorie:** bug · **Gebied:** booking-payments
- **Locatie:** `app/(site)/boeken/actions.js:147`
- **Impact:** A member who applied a valid discount code, abandoned the first checkout, then clicked 'Afrekenen'/'Betaal nu' from the account page or pending banner is charged the full €15 instead of the discounted amount — and per BK-06 their code is already marked used so they cannot re-enter it. Member is overcharged.
- **Fix:** Persist the agreed charge amount (and discount_code_id) on the booking when the first checkout is created, then have resumeCheckoutAction reuse that stored amount/code in checkoutParams. Combined with BK-06 (record redemption on payment), this keeps the resumed price consistent with the original quote.

### 27. Assigning a program sets member_id on the shared template instead of copying — second assignment steals it from the first member
- **Categorie:** data-integrity · **Gebied:** coach-domain
- **Locatie:** `app/coach/coaching-actions.js:121-135`
- **Impact:** A 'template' can belong to only one member at a time. If a coach builds one template and assigns it to member A, then later assigns the same program to member B, member A's program silently disappears (member_id is overwritten to B). There is no COPY step, so the intended 'assign = copy then per-member editable' behavior is broken: editing a template after assignment also live-edits the assigned member's schedule, and you cannot give the same template to multiple clients.
- **Fix:** On assign, clone the program: insert a new programs row (coach_id, member_id, is_template=false) plus copies of all program_days and program_exercises, then notify. Keep the original as a reusable template (is_template stays true). Alternatively, when memberId is set, deep-copy via SQL function so each member gets an independent, editable program.

### 28. coach_book_session ignores admin slot blocks and skips the persons<=capacity check the member RPC enforces

### 29. setClientPrice silently writes €0 when the price field is blank or non-numeric
- **Categorie:** data-integrity · **Gebied:** coach-domain

### 40. CRON_SECRET is passed in the URL query string on internal self-trigger fetches
- **Categorie:** security · **Gebied:** api-security
- **Locatie:** `app/api/cron/activation/route.js:30`
- **Impact:** The cron and queue chain re-invoke the worker with the secret in the query string (`?key=<CRON_SECRET>`). Query strings are routinely logged by Vercel/edge access logs, proxies, and analytics, and end up in the Referer header — leaking the long-lived shared secret that protects the service-role-backed cron/queue endpoints. Same leak at app/api/queue/process/route.js:30 and app/beheer/newsletter-actions.js:13. The endpoints even accept the secret via `searchParams.get('key')` (route.js:18), encouraging this leak.
- **Fix:** Pass the secret only in the Authorization header on the internal fetch: fetch(url, { headers: { Authorization: `Bearer ${secret}` } }) and remove the `?key=` query-param acceptance from both route handlers.

### 41. Inbound email importer hardcodes the oldest gym instead of routing by recipient
- **Categorie:** data-integrity · **Gebied:** api-security
- **Locatie:** `app/api/inbound/email/route.js:42-43`
- **Impact:** Every inbound @fittin.be email is attributed to whichever gym was created first, regardless of which gym the message was actually addressed to. The codebase is explicitly multi-tenant (gym_id on every table, per project conventions). Once a second gym exists, all inbound mail lands in the first gym's inbox — a cross-tenant data leak and misrouting bug. importReceived itself takes a gymId param, so the routing decision is made here incorrectly.
- **Fix:** Resolve the gym from the recipient address: fetch the message's `to` address (or pass it through) and look up the gym by its configured inbound/contact address (e.g. a gyms.inbound_email or domain mapping) instead of `order('created_at').limit(1)`.
